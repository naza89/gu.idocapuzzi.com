05 — Gaps: qué falta exactamente para cerrar el flujo de pagos
Componente
Situación actual
Qué necesita Antigravity
Prioridad
nave_payment_id, nave_status
en tabla ordenes
No existen en el schema. La tabla ordenes no tiene ningún campo de NAVE.
Ejecutar SQL:
ALTER TABLE ordenes ADD COLUMN nave_payment_id TEXT; ADD COLUMN nave_status VARCHAR(30); ADD COLUMN nave_monto_ars DECIMAL(12,2);
CRÍTICO
Sin esto, el webhook no puede actualizar la orden.
POST /api/nave/crear-pago
No existe. El frontend lo llama y queda colgado.
Autenticar con NAVE Sandbox → crear payment_request → UPDATE ordenes (estado: pago_pendiente, nave_payment_id) → devolver { payment_request_id, qr_data }.
CRÍTICO
Bloquea todo el Step 3.
POST /api/webhooks/nave
No existe. NAVE no tiene dónde enviar la notificación.
Responder 200 inmediato → async: GET payment_check_url → UPDATE ordenes (estado, nave_status, pagado_at si APPROVED).
CRÍTICO
Sin esto, el pago nunca se confirma en Supabase.
UPDATE ordenes al confirmar envío
(Step 2 → Step 3)
El frontend sabe qué envío eligió el usuario pero no actualiza la orden en Supabase con esos datos antes de pasar al Step 3.
Agregar llamada al backend en start.js al confirmar el envío: PATCH /api/ordenes/{id} con tipo_envio, precio_envio, id_sucursal_oca.
IMPORTANTE
La orden queda incompleta si no se persiste.
SUPABASE_SERVICE_ROLE_KEY
El backend necesita esta key para hacer UPDATE en ordenes desde server-side (las políticas RLS actuales permiten UPDATE con anon key, pero en producción conviene service role).
Agregar en variables de entorno de Vercel. No exponer en frontend.
IMPORTANTE
Reducir stock al confirmar pago
No existe ningún mecanismo para decrementar stock en variantes_producto cuando un pago se aprueba.
En el webhook APPROVED: para cada item_orden, UPDATE variantes_producto SET stock = stock - cantidad WHERE id = variante_id.
IMPORTANTE
Sin esto el stock no refleja las ventas.
POST /api/oca/crear-envio
No existe. OCA ePak solo tiene el cotizador implementado hasta ahora.
Solo se dispara después de APPROVED. Toma los datos de la orden (calle, numero, ciudad, CP) y los manda a OCA para generar el envío y obtener el nro_envio_oca.
FASE 2
No bloquea el pago.
Carga de peso/medidas OCA
en variantes_producto
Los campos peso_kg, alto_cm, ancho_cm, largo_cm ya existen (07_oca_envios.sql) pero están vacíos. OCA usa estimados hoy.
Completar manualmente o con un script los pesos y medidas de las prendas para cotizaciones más precisas.

06 — SQL listo para ejecutar: agregar columnas NAVE a la tabla ordenes
08_agregar_nave.sql — Ejecutar en Supabase Dashboard › SQL Editor
-- NAVE payment tracking columns
ALTER TABLE ordenes ADD COLUMN IF NOT EXISTS nave_payment_id  TEXT;
ALTER TABLE ordenes ADD COLUMN IF NOT EXISTS nave_status      VARCHAR(30);
ALTER TABLE ordenes ADD COLUMN IF NOT EXISTS nave_monto_ars   DECIMAL(12,2);

COMMENT ON COLUMN ordenes.nave_payment_id IS 'ID del pago en NAVE (payment_request_id de Ranty)';
COMMENT ON COLUMN ordenes.nave_status     IS 'Estado NAVE: PENDING, APPROVED, REJECTED, CANCELLED';
COMMENT ON COLUMN ordenes.nave_monto_ars  IS 'Monto cobrado por NAVE en ARS (para conciliación)';

-- Índice para búsquedas por payment ID desde webhook
CREATE INDEX IF NOT EXISTS idx_ordenes_nave_payment_id ON ordenes(nave_payment_id);

-- Estados de orden actualizados con los nuevos valores NAVE
COMMENT ON COLUMN ordenes.estado IS 'pendiente → envio_calculado → pago_pendiente → pagado → preparando → enviado → entregado | cancelado';
